## 스킬 개요

이 스킬은 **`/address-review-rust` 커맨드가 입력될 때 자동으로 실행**된다.
코드 리뷰를 받은 당사자가 리뷰 지적 사항을 처리할 때 Claude가 다음을 수행한다:

1. 리뷰 코멘트 수집 (대화 이력 또는 PR API)
2. 각 지적의 타당성 독립 평가
3. 타당한 지적만 수정 적용 (Before/After 제시 → 인간 확인)
4. 대응하지 않은 지적에 대한 근거 명시
5. 최종 요약 출력

**핵심 원칙:**
- **독립 판단** — 리뷰어의 지적을 맹목적으로 따르지 않고 기술적 타당성을 독자적으로 평가한다
- **근거 명시** — 대응하지 않을 경우 리뷰어가 납득할 수 있는 이유를 반드시 기재한다
- **안전한 수정** — 수정으로 새로운 문제가 발생하지 않는지 cargo test + clippy로 확인한다
- **인간 확인** — 각 수정 전 Before/After를 제시하고 승인 후에만 적용한다

## 커맨드 문법

```
/address-review-rust 42           PR #42의 리뷰·인라인 코멘트를 대상으로 대응 ← 권장
/address-review-rust              직전 대화의 리뷰 내용을 대상으로 대응 (같은 세션 전용)
```

> **⚠️ 권장**: 리뷰어가 `/code-review-feedback-rust`로 GitHub에 게시한 코멘트를 대응할 때는
> 반드시 **PR 번호를 명시**(`/address-review-rust [번호]`)해야 한다.
> 인수 없는 대화 모드는 같은 세션에서 `/code-review-rust`를 실행한 경우에만 정상 동작한다.


## [대화 모드] STEP 0-CONV — 직전 대화에서 리뷰 내용 추출

**인수 없이 `/address-review-rust`만 실행했을 때 이 단계를 따른다.**

Claude가 직전 대화 이력에서 리뷰 내용을 자동으로 특정한다:

**추출 기준:**
- `/code-review-rust` 실행 결과 리포트
- 분석 리포트 내 이슈 항목 (`🔴 Critical`, `🟠 High`, `🟡 Medium`, `🔵 Low`)
- 수정 계획 목록 (STEP 3 출력물)

**추출 후 출력 형식:**

```

📥  리뷰 내용 추출 완료 (대화 이력 기준)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
출처:    직전 /code-review-rust 결과
지적 수: [N]건 (Critical [N] / High [N] / Medium [N] / Low [N])
대상:    [리뷰된 파일 또는 모드 정보]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## [PR 모드] STEP 0-PR — PR 리뷰·코멘트 수집

**PR 번호를 지정했을 때 이 단계를 따른다.**

Claude가 Bash 도구로 아래 커맨드를 직접 실행하여 코멘트를 수집한다:

```bash

gh api repos/{owner}/{repo}/pulls/{번호}

# 2. 리뷰 본문 수집 (전체 리뷰 코멘트)
gh api repos/{owner}/{repo}/pulls/{번호}/reviews

# 3. 인라인 코드 코멘트 수집
gh api repos/{owner}/{repo}/pulls/{번호}/comments
```

**수집 후 출력 형식:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📥  PR #[번호] 리뷰 수집 완료
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
제목:      [PR 제목]
리뷰어:    [리뷰어 목록]
리뷰 수:   [N]건 (승인 [N] / 변경 요청 [N] / 코멘트 [N])
인라인:    [N]개 파일, [N]개 코멘트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

| 심각도 | 리뷰어 명시 또는 문맥 추정 (Critical / High / Medium / Low) |

## STEP 2 — 각 지적의 타당성 평가

Claude가 각 지적에 대해 아래 3가지 기준으로 독립적으로 평가한다.
**리뷰어 의견에 동조하지 않고 코드와 프로젝트 정책을 직접 확인하여 판단한다.**

### 평가 기준

**① 기술적 타당성**
- 지적된 코드가 실제로 버그·성능·보안 문제를 일으키는가?
- Rust 언어 규칙 및 관용 표현에 비추어 맞는 지적인가?
- `CODE_REVIEW_RUST.md`의 C-CR-01~C-CR-10 카테고리 기준에 부합하는가?

**② 프로젝트 정책 적합성**
- `security.md`, `test.md` 규칙과 충돌하지 않는가?
- `CLAUDE.md`의 코딩 컨벤션(에러 처리, 소유권, 타입 설계 등)에 맞는가?
- 현재 프로젝트 맥락(레이어 역할, 의존성 구조)을 고려할 때 적절한가?

**③ 구현 트레이드오프 타당성**
- 수정 시 다른 코드에 미치는 영향 범위는 적절한가?
- 성능·가독성·유지보수성 간 균형이 맞는가?
- 수정 비용 대비 실질적 개선 효과가 있는가?

### 평가 결과 분류

| 판정 | 의미 | 처리 |
|------|------|------|
| ✅ 대응 | 기술적으로 타당하고 수정이 필요하다 | STEP 3에서 수정 |
| ⚠️ 부분 대응 | 방향은 맞지만 제안 방식과 다르게 수정한다 | STEP 3에서 대안 수정 |
| ❌ 대응 불필요 | 타당하지 않거나 현재 프로젝트에 적합하지 않다 | 이유 명시 후 스킵 |

### 평가 결과 출력 형식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🧠  타당성 평가 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 대응    [N]건: [A-RV-XX], [A-RV-XX] …
⚠️ 부분    [N]건: [A-RV-XX] …
❌ 불필요  [N]건: [A-RV-XX] …

[A-RV-XX] ✅ 대응
  근거: [C-CR-XX 위반 / security.md §섹션 해당 / 실제 버그 가능성 등]

[A-RV-XX] ❌ 대응 불필요
  근거: [기술적으로 틀린 이유 / 프로젝트 정책과 불일치 / 트레이드오프 판단 등]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 3 — 수정 적용 (Before/After → 인간 확인 → 커밋)

**대응 / 부분 대응** 판정을 받은 지적에 대해서만 이 단계를 수행한다.
Claude는 **절대 먼저 코드를 변경하지 않는다.** Before/After를 제시하고 승인 후에만 적용한다.

### 수정 제안 출력 형식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[판정 이모지] [A-RV-XX] [지적 제목]  —  수정 제안
    ([진행 현황: N/M번째])
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 위치:   [파일명 : 행번호]
📖 지적:   [리뷰어 원문 요약]
🧠 판정:   [✅ 대응 / ⚠️ 부분 대응] — [판정 근거 1~2줄]

─── BEFORE ──────────────────────────────
[현재 코드]

─── AFTER ───────────────────────────────
[수정 코드]

─── 수정 근거 ───────────────────────────
  • [변경 포인트 1]
  • [변경 포인트 2]
  • (⚠️ 부분 대응 시) 리뷰어 제안과 다른 이유: [설명]

─── 검증 커맨드 ─────────────────────────
  cargo fmt
  cargo clippy -- -D warnings
  cargo test [관련_테스트_경로]

─── 커밋 메시지 제안 ────────────────────
  fix([scope]): [A-RV-XX] [50자 이내 요약]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
👆 이 수정을 적용할까요?

   ✅ "적용" / "ok"          → 수정 적용 후 다음 지적
   ❌ "건너뜀" / "skip"      → 이 지적은 대응 불필요로 전환
   ✏️  "수정해줘: [요청]"    → After 코드 재제안
   ⏸️  "여기서 멈춰"          → 완료 요약으로 이동
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 사용자 응답별 처리

| 응답 | Claude 행동 |
|------|-------------|
| `"적용"` / `"ok"` / `"ㅇ"` | Edit 도구로 코드 수정 + cargo test 실행 + 다음 지적 |
| `"건너뜀"` / `"skip"` / `"ㄴ"` | 대응 불필요 목록에 추가 → 다음 지적 |
| `"수정해줘: [내용]"` | After 코드 재제안 → 재출력 |
| `"왜?"` / `"설명해줘"` | 판정 근거 상세 설명 → 같은 제안 유지 |
| `"전체 적용"` | 대응 판정 전체 일괄 적용 |
| `"여기서 멈춰"` / `"stop"` | 루프 종료 → STEP 4 |

### 수정 적용 후 검증

각 수정 적용 직후 Claude가 Bash 도구로 직접 실행한다:

```bash
cargo fmt
cargo clippy -- -D warnings
cargo test 2>&1 | tail -5
```

검증 실패 시 자동으로 수정을 롤백하고 원인을 분석하여 재제안한다.

## STEP 4 — 최종 요약 출력

모든 지적 처리 완료 후 아래 형식으로 요약을 출력한다.

````markdown
## 리뷰 대응 요약

### 지적 목록
- [x] [A-RV-01] [지적 개요] — ✅ 대응 완료
- [x] [A-RV-02] [지적 개요] — ⚠️ 부분 대응 완료
- [ ] [A-RV-03] [지적 개요] — ❌ 대응 불필요

### 대응한 지적
| 지적 | 판정 | 수정 내용 |
|------|------|----------|
| [A-RV-01] [제목] | ✅ 대응 | [수정 개요] |
| [A-RV-02] [제목] | ⚠️ 부분 | [수정 개요 + 리뷰어 제안과 다른 점] |

### 대응하지 않은 지적
| 지적 | 이유 |
|------|------|
````

## 브랜치 및 커밋 정책

**브랜치 네이밍:**
```
PR 대응 시    → fix/cr-pr[번호]-[module]   (예: fix/cr-pr42-order)
대화 대응 시  → 현재 작업 브랜치에 그대로 커밋
```

**커밋 메시지 형식:**
```
fix([scope]): [A-RV-XX] [50자 이내 요약]

예시:
  fix(order): [A-RV-01] unwrap() → Result 전파로 교체
  fix(auth):  [A-RV-03] std::Mutex → tokio::Mutex 교체
```

## 주의사항

- 리뷰어의 권위에 의존하지 않고 **코드와 정책을 직접 검토하여** 판단한다
- 대응하지 않는 경우, 리뷰어가 납득할 수 있는 **기술적 근거를 반드시** 기재한다
- 수정으로 인해 새로운 문제(컴파일 에러, 테스트 실패, clippy 경고)가 발생하지 않도록 매 수정 후 검증한다
- `security.md`, `test.md` 규칙과 충돌하는 리뷰어 제안은 규칙을 우선한다
- PR 모드에서 모든 수정 커밋 완료 후 **반드시 `git push origin [브랜치명]`** 을 안내한다
- push 완료 후 `/reply-review-rust [PR번호]`로 각 리뷰 코멘트에 대응 회신한다
