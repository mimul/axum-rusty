# `/skill-tunning` 커맨드 스킬

## 스킬 개요

이 스킬은 **`/skill-tunning <커맨드명>` 형식으로 실행**된다.
지정한 커맨드의 스킬 마크다운 파일(SKILL.md 및 참조 파일)을 읽고,
**고정 테스트 케이스 5개 × 5개 평가 기준** 체계로 현재 프롬프트를 채점한 뒤,
**단일 변수 수정 원칙**에 따라 반복 개선안을 제안한다.

**핵심 운영 규칙**:
- **점수로 판단** — 감이 아닌 Evaluation Criteria 5항목 × 각 5점 = 25점 만점
- **단일 변수** — 이터레이션마다 정확히 하나의 수정만 적용
- **테스트 고정** — 튜닝 내내 동일한 5개 테스트 케이스 사용
- **보여주고 확인받기** — Before/After 제시 → 승인 후에만 파일 저장
- **최소 3회 반복** — 점수가 22점 이상이거나 사용자가 중단 요청 시 종료

## 커맨드 문법

```
/skill-tunning <커맨드명>          지정 커맨드 스킬 튜닝 (기본: 전체 이터레이션)
/skill-tunning <커맨드명> --eval   채점만 수행 (파일 수정 없음)
/skill-tunning <커맨드명> --iter N N회 이터레이션만 수행 (기본: 3)
/skill-tunning --list              등록된 커맨드 목록 출력
/skill-tunning --help              사용법 출력
```

**등록된 커맨드** (인자로 사용 가능):

| 커맨드명 | 스킬 경로 |
|----------|-----------|
| `code-review-rust` | `.claude/skills/code-review-rust/SKILL.md` |
| `code-review-feedback-rust` | `.claude/skills/code-review-feedback-rust/SKILL.md` |
| `refactor-rust` | `.claude/skills/refactor-rust/SKILL.md` |
| `address-review-rust` | `.claude/skills/address-review-rust/SKILL.md` |
| `reply-review-rust` | `.claude/skills/reply-review-rust/SKILL.md` |
| `test-rust` | `.claude/skills/test-rust/SKILL.md` |
| `skill-tunning` | `.claude/skills/skill-tunning/SKILL.md` |

## 실행 흐름

```
STEP 0  →  STEP 1  →  STEP 2  ─→  STEP 3  ──┐
커맨드    스킬 파일   테스트 케이스   초기 채점  │
검증      로드        설계 (고정)    + 실패 패턴  │
                                               ↓
                                         STEP 4 (× 최소 3회)
                                         수정안 Before/After 제시
                                         ↓ [승인]
                                         파일 저장
                                         ↓ [다음 이터레이션]
                                        STEP 5
                                        최종 채점 + 스코어 트렌드
                                        + 개선된 SKILL.md 확정
```

**STEP 0~2는 자동 실행. STEP 4 각 이터레이션과 STEP 5에서 인간 응답을 기다린다.**

## STEP 0 — 커맨드 검증 및 파일 경로 확정

Claude가 Bash 도구로 직접 실행하며, 사용자 확인 없이 자동으로 진행한다.

```bash
# 1. 스킬 디렉토리 존재 여부 확인
ls .claude/skills/<커맨드명>/

# 2. SKILL.md 위치 확인
ls .claude/skills/<커맨드명>/SKILL.md

# 3. 참조 파일 목록 파악 (예: CODE_REVIEW_RUST.md, TEST_RUST.md 등)
ls .claude/skills/<커맨드명>/
```

**검증 결과별 분기:**

| 상태 | 처리 |
|------|------|
| SKILL.md 존재 | 경로 확정 후 즉시 STEP 1 진행 |
| 커맨드명 불일치 | 등록된 커맨드 목록 출력 + 재입력 요청 |
| 디렉토리 없음 | 오류 출력 후 종료 |

**출력 형식:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯  /skill-tunning 준비 완료
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
대상 커맨드: /<커맨드명>
SKILL.md:   .claude/skills/<커맨드명>/SKILL.md
참조 파일:  [발견된 파일 목록]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 1 — 스킬 파일 로드

Claude가 Read 도구로 직접 읽는다. 사용자에게 내용 붙여넣기를 요청하지 않는다.

```
Read(file_path: ".claude/skills/<커맨드명>/SKILL.md")
Read(file_path: ".claude/skills/<커맨드명>/<참조파일>.md")  # 존재하는 경우
```

읽은 후 내부적으로 파악하는 항목:
- 스킬이 다루는 모드 수 및 목록
- STEP 구조 존재 여부
- 출력 포맷 정의 명확성
- 금지 사항(Anti-Pattern) 정의 여부
- 인간 확인 루프 존재 여부

## STEP 2 — 고정 테스트 케이스 설계

테스트 케이스는 **대상 커맨드의 실제 사용 시나리오** 기반으로 설계한다.
튜닝 기간 내내 변경 금지.

### 테스트 케이스 구성 원칙

| 유형 | 수 | 목적 |
|------|----|------|
| Normal Case | 2개 | 가장 일반적인 호출 시나리오 |
| Edge Case | 2개 | 경계 조건, 불완전한 입력 |
| Failure Case | 1개 | 스킬이 실패·오작동할 만한 입력 |

### 커맨드별 테스트 케이스 예시

**`code-review-rust` 테스트 케이스:**

```
TC-01 [Normal]  /code-review-rust — staged + unstaged 변경 있는 경우
TC-02 [Normal]  /code-review-rust --pr 42 — PR 리뷰 요청
TC-03 [Edge]    /code-review-rust — 변경사항 없음 (working tree clean)
TC-04 [Edge]    /code-review-rust --pr 999 — 존재하지 않는 PR 번호
TC-05 [Failure] /code-review-rust src/main.rs — unsafe 블록 + SAFETY 주석 없음
```

**`test-rust` 테스트 케이스:**

```
TC-01 [Normal]  /test-rust — 전체 코드베이스 커버리지 분석
TC-02 [Normal]  /test-rust --type db — Repository DB 테스트만
TC-03 [Edge]    /test-rust — 이미 모든 테스트가 작성된 경우
TC-04 [Edge]    /test-rust usecase/nonexistent.rs — 존재하지 않는 파일
TC-05 [Failure] /test-rust — testcontainers 없이 DB 테스트 요청
```

테스트 케이스 설계 후 아래 형식으로 출력하고 사용자 확인을 받는다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋  테스트 케이스 설계 (고정 — 튜닝 내내 불변)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TC-01 [Normal]   <입력>
  기대 동작: <스킬이 해야 할 행동>
  평가 포인트: <무엇을 보는가>

TC-02 [Normal]   <입력>
  기대 동작: ...

TC-03 [Edge]     <입력>
  기대 동작: ...

TC-04 [Edge]     <입력>
  기대 동작: ...

TC-05 [Failure]  <입력>
  기대 동작: <이 경우 스킬이 어떻게 처리해야 하는가>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"확정" / "케이스 수정: [번호] [내용]"
```

## STEP 3 — 초기 채점 (Iteration 0)

고정 테스트 케이스 × 5개 평가 기준으로 현재 SKILL.md를 채점한다.

### Evaluation Criteria (0~5점 each, 합계 25점)

| 기준 | 설명 |
|------|------|
| **Accuracy** (정확성) | 각 TC에서 스킬이 요구사항을 정확히 수행하는가 |
| **Completeness** (완전성) | 모드·엣지 케이스 처리 등 누락된 기능이 없는가 |
| **Robustness** (안정성) | 예외 입력·실패 케이스에 적절히 대응하는가 |
| **Clarity** (명확성) | 출력 포맷·지시문이 모호함 없이 명확한가 |
| **Constraint Compliance** (제약 준수) | CLAUDE.md·rules/*.md의 규칙을 준수하는가 |

### 채점 리포트 형식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊  초기 채점 (Iteration 0)  —  /<커맨드명>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

| 기준                    | TC-01 | TC-02 | TC-03 | TC-04 | TC-05 | 평균 |
|-------------------------|-------|-------|-------|-------|-------|------|
| Accuracy                |  X/5  |  X/5  |  X/5  |  X/5  |  X/5  | X.X  |
| Completeness            |  X/5  |  ...  |       |       |       | X.X  |
| Robustness              |  X/5  |  ...  |       |       |       | X.X  |
| Clarity                 |  X/5  |  ...  |       |       |       | X.X  |
| Constraint Compliance   |  X/5  |  ...  |       |       |       | X.X  |
| **Total**               | XX/25 |       |       |       |       | **XX.X/25** |

🚨 실패 패턴:
  • [패턴 유형] — TC-[번호]: [구체적인 문제]
    → 원인: Missing requirement / Wrong logic / Format violation / Hallucination

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## STEP 4 — 이터레이션 루프 (최소 3회)

한 번에 **정확히 하나의 수정**만 제안한다.
Claude는 절대 먼저 파일을 변경하지 않는다.

### 4-A. 수정 유형 결정 (우선순위 순)

| 우선순위 | 수정 유형 | 적용 조건 |
|----------|-----------|-----------|
| 1 | **명확화 (clarify)** | 지시문이 모호하여 TC에서 오작동 |
| 2 | **제약 추가 (constraint)** | 금지 행동이 명시되지 않아 실패 케이스 발생 |
| 3 | **예시 추가 (example)** | 올바른 동작을 예시 없이 설명만으로 전달 |
| 4 | **구조 변경 (structure)** | STEP 순서·섹션 배치가 비효율적 |

### 4-B. Before/After 비교 출력 형식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔧  Iteration [N] — 수정 제안
    (수정 유형: [clarify / constraint / example / structure])
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📍 수정 위치:   SKILL.md [섹션명 / 행 범위]
📖 문제:        [어떤 TC에서 어떻게 실패했는가 — 1~2줄]
🏷️  수정 유형:  [유형] — 한 번에 하나만
📏 근거 TC:    TC-[번호] [Normal/Edge/Failure]

─── BEFORE ──────────────────────────────
[수정 전 SKILL.md 해당 섹션]

─── AFTER ───────────────────────────────
[수정 후 SKILL.md 해당 섹션]

─── 예상 점수 변화 ──────────────────────
  현재: XX/25
  예상: YY/25  (+Z점)
  개선 기준: [어떤 Evaluation Criteria가 올라가는가]

─── 다른 TC 영향 ────────────────────────
  [이 수정이 다른 TC에 미치는 영향 — 부작용 없으면 "없음"]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
👆 이 수정을 적용할까요?

   ✅ "적용" / "ok" / "ㅇ"   → SKILL.md 저장 후 다음 이터레이션 채점
   ❌ "건너뜀" / "skip"       → 두 번째 우선순위 수정으로 대체 제안
   ✏️  "수정해줘: [요청]"     → After 재제안
   ⏸️  "여기서 멈춰"           → STEP 5로 이동
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 4-C. 사용자 응답별 처리

| 응답 | Claude 행동 |
|------|-------------|
| `"적용"` / `"ok"` / `"ㅇ"` | SKILL.md 저장 → 재채점 → 이터레이션 카운터 +1 |
| `"건너뜀"` / `"skip"` | 다음 우선순위 수정 유형으로 대체 제안 |
| `"수정해줘: [내용]"` | After 재제안 → 동일 형식 재출력 |
| `"왜?"` / `"설명해줘"` | TC별 실패 상세 설명 → 동일 Before/After 유지 |
| `"여기서 멈춰"` / `"stop"` | 루프 종료 → STEP 5 |

### 4-D. 적용 후 재채점 출력 형식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅  Iteration [N] 적용 완료 — 재채점
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

변경 파일: .claude/skills/<커맨드명>/SKILL.md

| 기준                  | 이전 | 이후 | 변화 |
|-----------------------|------|------|------|
| Accuracy              | X.X  | Y.Y  |  +Z  |
| Completeness          | X.X  | Y.Y  |  +Z  |
| Robustness            | X.X  | Y.Y  |  +Z  |
| Clarity               | X.X  | Y.Y  |  +Z  |
| Constraint Compliance | X.X  | Y.Y  |  +Z  |
| **Total**             | XX/25| YY/25| **+Z** |

잔여 실패 패턴: [남은 문제 목록 또는 "없음"]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 4-E. 종료 조건

| 조건 | 처리 |
|------|------|
| 총점 ≥ 22점 AND 이터레이션 ≥ 3 | 자동 종료 → STEP 5 |
| 이터레이션 수 = `--iter N` 도달 | 종료 → STEP 5 |
| 사용자 `"여기서 멈춰"` | 즉시 STEP 5 |
| 3회 연속 동일 점수 (정체) | 경고 후 사용자에게 선택 요청 |

---

## STEP 5 — 최종 채점 및 개선 요약 출력

### 5-A. 스코어 트렌드 및 최종 리포트

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎉  튜닝 완료 — /<커맨드명> SKILL.md
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📈 Score Trend:
  Iteration 0 (초기): XX/25
  Iteration 1:        XX/25  (+Z)  [clarify: 섹션명]
  Iteration 2:        XX/25  (+Z)  [constraint: 섹션명]
  Iteration 3:        XX/25  (+Z)  [example: 섹션명]
```

## Anti-Pattern (금지)

```
🚫 한 이터레이션에 여러 섹션 동시 수정 — 단일 변수 원칙 위반
🚫 테스트 케이스 도중 변경 — 기준이 흔들리면 점수 비교 불가
🚫 평가 기준 임의 조정 — 5개 기준 × 0~5점 고정
🚫 점수를 올리려 AFTER에 내용을 과도하게 추가 — 간결성 훼손
🚫 수정 제안 없이 "충분히 좋다"고 판단 — 최소 3회 이터레이션 필수
🚫 CLAUDE.md 금지 사항(unsafe, cargo publish 등) 완화하는 방향 수정
🚫 사용자 승인 없이 SKILL.md 파일 저장
```
