# Empirical Prompt Tuning

프롬프트의 품질은 작성자가 판단할 수 없다.
작성자가 “명확하다”고 느끼는 지시일수록, 다른 에이전트는 쉽게 막힌다.

이 방법론의 핵심은 다음이다:

> **작성자가 평가하지 않는다. 반드시 subagent가 실행한다.**

그리고 그 결과를 기반으로 **반복 개선**한다.

---

## 커맨드 문법

```
/prompt-tunning <스킬명>           지정 스킬 튜닝 (기본: 최소 3회 이터레이션)
/prompt-tunning <스킬명> --eval    채점만 수행 (파일 수정 없음)
/prompt-tunning <스킬명> --iter N  N회 이터레이션만 수행
```

**스킬명**: `.claude/skills/<스킬명>/SKILL.md` 또는 `.claude/commands/<스킬명>.md` 기준으로 탐색한다.

---

## 언제 사용하는가

* 프롬프트 / 스킬 신규 작성 직후
* 대규모 수정 이후
* 에이전트가 기대대로 동작하지 않을 때
* 핵심 자동화 프롬프트를 안정화할 때

---

## 사용하지 않는 경우

* 일회성 프롬프트
* 단순 문장 스타일 수정
* 주관적 취향 반영

---

## 워크플로우

### 0. 정합성 체크 (필수)

* description과 body가 일치하는지 확인
* 불일치 시 수정 후 진행

---

### 1. 베이스라인 준비

#### 시나리오

* 2~3개

  * 일반 케이스 1
  * 엣지 케이스 1~2

#### 체크리스트

* 3~7개
* 반드시 `[critical]` 1개 이상 포함
* 사전 정의 후 변경 금지

---

### 2. Subagent 실행

* 반드시 Task tool 사용
* 매 iteration마다 새로운 subagent 생성
* 동일 agent 재사용 금지

---

### 3. 실행

subagent에게 다음을 전달:

* 대상 프롬프트
* 시나리오
* 체크리스트

**Subagent 프롬프트 템플릿:**

```
다음 스킬 프롬프트를 시나리오에 따라 실행하고, 반드시 자기보고를 작성해라.
코드나 파일을 실제로 수정하지 말고, 어떻게 진행할지 판단하고 보고만 해라.

=== 스킬 프롬프트 ===
[SKILL.md 전체 내용]

=== 실행 시나리오 ===
[시나리오 설명]

=== 자기보고 (필수) ===

1. 불명확했던 지시
   - 어느 부분이, 왜 불명확했는가 / 없었다면 "없음"

2. 임의로 판단한 사항
   - 무엇을 어떻게 판단했는가 / 없었다면 "없음"

3. 막혔던 지점
   - 어디서, 왜 막혔는가 / 없었다면 "없음"

4. 체크리스트 충족 여부
   [critical] <항목>: ○ / 부분 / ×
   <항목>: ○ / 부분 / ×

5. 전체 수행 난이도
   쉬움 / 보통 / 어려움 — 이유 한 줄
```

---

### 4. 평가 (핵심)

#### 4-1. 실행자 자기보고

* 불명확한 부분
* 임의 판단
* 막힌 지점

#### 4-2. 메트릭 (자기보고 기반)

| 항목        | 설명                    |
| --------- | --------------------- |
| 성공/실패     | critical 전부 충족 여부     |
| 정확도       | 체크리스트 충족률             |
| 불명확점 수   | 자기보고의 불명확 항목 수 (0이 최선) |
| 임의 판단 수  | 자기보고의 임의 판단 항목 수 (0이 최선) |
| 수행 난이도   | 쉬움 / 보통 / 어려움        |

---

### 성공 기준

* `[critical]` 항목 전부 충족 → 성공
* 하나라도 실패 → 실패

---

### 정확도 계산

* ○ = 1
* 부분 = 0.5
* × = 0

---

### 5. 수정

❌ 금지

* “더 명확하게 수정”
* “설명을 보강”
* 사용자 승인 없이 파일 저장

✅ 필수

* 어떤 체크리스트 항목을 개선하는지 명시
* 수정 전 Before/After를 제시하고 사용자 승인을 받은 후에만 파일을 저장

---

### 6. 반복

* 동일 시나리오 유지
* 최소 수정 적용
* 항상 새로운 subagent 사용
* 아래 **수렴 조건** 2회 연속 충족 시 종료

---

## Iteration 기록 템플릿

```
## Iteration N

### 변경점
- ...

### 결과
| 시나리오 | 성공 | 정확도 | 불명확점 | 임의 판단 | 난이도 |
|---|---|---|---|---|---|

### 불명확점
- ...

### 임의 판단
- ...

### 다음 수정
- ...
```

---

## 수렴 조건

2회 연속:

* 신규 불명확점 0
* 정확도 개선 ≤ 3%
* 신규 임의 판단 0

---

## 실패 패턴

* subagent 미사용
* 동일 agent 재사용
* 시나리오 변경
* 자기보고 수집 생략

---

## 환경 제약

Task tool 사용 불가 시:

```
empirical evaluation skipped: dispatch unavailable
```

자기 평가 금지

---

## 핵심 요약

1. subagent 실행
2. 자기보고 수집
3. 체크리스트 평가
4. 최소 수정 반복

이 4가지를 지키지 않으면 이 방법론은 무효다.
