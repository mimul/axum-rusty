# 도메인 중심 진화형 코딩 스타일

## 1. 핵심 원칙 (Principles)

### 1.1 변화 용이성 우선
- 성능보다 **변경하기 쉬운 코드**를 우선한다
- 미래 요구사항을 추측하여 설계하지 않는다
- 추상화는 최소 3번의 반복 이후에 도입한다 (Rule of Three)

### 1.2 의도를 드러내는 코드
- 코드는 “무엇을 하는지”가 아니라 **왜 존재하는지**를 설명해야 한다
- 짧은 코드보다 **이해하기 쉬운 코드**를 선택한다
- 주석보다 코드 구조 개선을 우선한다

### 1.3 도메인 중심 설계
- 도메인 개념이 코드 구조를 결정해야 한다
- 유비쿼터스 언어를 코드 전반에 일관되게 사용한다
- 도메인 없이 범용 유틸리티를 먼저 만들지 않는다

### 1.4 점진적 진화
- 항상 작고 안전한 단위로 변경한다
- 한 번의 변경에는 하나의 의도만 담는다
- 모든 변경은 되돌릴 수 있어야 한다

### 1.5 구현하기 전에 생각할 것
- 불확실한것, 모호성이 있을 경우, 불문명한 것이 있을 경우 추측보다는 질문이나 설명을 요청하세요.
- 더 간단한 방식이 있을 경우 언급해 주세요.

## 2. 설계 규율 (Design Discipline)

### 2.1 구조
- 작고 조합 가능한 함수로 구성한다
- 함수는 하나의 책임만 가진다
- 조건문/중첩 깊이는 최대 2단계로 제한한다

### 2.2 도메인 모델
- 빈약한 도메인 모델(Anemic Model)을 지양한다
- 데이터와 행동을 함께 둔다
- 도메인 로직은 객체 내부에 캡슐화한다

### 2.3 추상화
- 중복 없이 추상화를 만들지 않는다
- 패턴이 드러난 이후에만 추상화한다
- 미래 확장을 위한 구조를 미리 만들지 않는다

### 2.4 제어 흐름
- 복잡한 표현보다 명확한 흐름을 선택한다
- 조건문은 의도를 드러내는 메서드로 치환한다
- Tell, Don’t Ask 원칙을 따른다

---

## 3. 리팩토링 철학

### 3.1 지속적인 리팩토링
- 리팩토링은 별도의 작업이 아니라 **개발 과정의 일부**
- 기능 추가 전, 필요하면 구조를 먼저 개선한다
- 항상 동작하는 상태를 유지한다

### 3.2 리팩토링 트리거
- 동일한 코드가 3번 이상 반복될 때
- 이름이 의도를 충분히 설명하지 못할 때
- 함수가 여러 책임을 가지기 시작할 때
- 도메인 개념이 코드에 드러나지 않을 때

---

## 4. 네이밍 규칙

### 4.1 기본 원칙
- 도메인 용어를 그대로 사용한다
- 축약보다 명확한 이름을 선택한다
- 구현 방식이 아니라 **의도**를 표현한다

### 4.2 금지
- `data`, `util`, `helper`, `manager` 같은 모호한 이름
- 불필요한 축약어
- 동일 개념에 대한 여러 이름 사용

---

## 5. 에지 케이스 & 경계 조건

### 5.1 기본 원칙
- 에지 케이스는 예외가 아니라 **도메인의 일부**로 취급한다
- 모든 경계 조건은 코드에 명시적으로 드러나야 한다
- 숨겨진 가정을 제거한다

### 5.2 모델링
- null / 없음 / 실패 상태를 명확한 타입이나 개념으로 표현한다
- invalid state는 가능한 한 타입 수준에서 제거한다
- 경계 조건을 조건문이 아닌 도메인 로직으로 표현한다

### 5.3 처리 방식
- 에러는 무시하지 않고 명확하게 전달하거나 처리한다
- fallback 로직은 반드시 의도를 드러낸다
- 임시 처리를 남기지 않는다

### 5.4 금지
- 침묵하는 실패
- 의미 없는 기본값
- 암묵적인 경계 처리

---

## 6. 성능 원칙 (Performance Principles)

### 6.1 기본 원칙
- 측정 없는 최적화는 하지 않는다 (premature optimization 금지)
- 가독성을 희생한 미세 최적화를 피한다
- 성능 문제는 병목을 먼저 측정하고 접근한다

### 6.2 알고리즘 & 복잡도
- O(n²) 이상의 복잡도는 명시적으로 정당화한다
- 반복문 내부에서 DB/네트워크 호출을 하지 않는다 (N+1 문제)
- 컬렉션 크기가 커질 수 있는 경우 페이지네이션을 기본으로 설계한다

### 6.3 데이터 접근
- 필요한 필드만 조회한다 (`SELECT *` 금지)
- 같은 데이터를 반복 조회하는 경우 캐싱 전략을 검토한다
- 트랜잭션 범위를 필요한 최소한으로 유지한다

### 6.4 리소스 관리
- 획득한 리소스(커넥션, 파일, 소켓)는 반드시 해제한다
- 불필요한 복사/복제를 피한다
- 공유 가변 상태는 최소화한다 — 동시성 문제의 주요 원인이다

### 6.5 금지
- 측정 없는 성능 최적화
- 가독성을 희생한 미세 최적화
- 리소스 미해제

---

## 7. 실행 규칙 (개발자 공통)

- 가독성을 최우선으로 한다
- 도메인을 명확히 모델링한다
- 변경하기 쉬운 구조를 유지한다
- 경계 조건을 명시적으로 처리한다
- 테스트로 검증한다
- 보안을 기본으로 고려한다
- 성능 문제는 측정 후 접근한다

---

## 8. 안티 패턴

- 성급한 추상화
- 빈약한 도메인 모델
- 깊은 중첩 구조
- 암묵적 경계 처리
- 테스트 없는 핵심 로직
- 보안을 고려하지 않은 설계
- 측정 없는 성능 최적화
- N+1 쿼리
